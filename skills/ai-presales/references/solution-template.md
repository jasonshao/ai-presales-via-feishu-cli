# Customer Solution Draft Template

The renderer (`render_solution.py`) emits this template populated with values from the context JSON. Agents are free to rewrite phrasing, but must keep the section structure so downstream tooling and reviewers can find each part predictably.

```markdown
# <Customer> x [Your Company] Solution Draft

> **Demo data — fictional customer.** This draft is generated by the Feishu CLI AI Presales Skill and must be reviewed by a human before sending to a customer.

## 1. Customer Background And Needs

### Business Context

<one paragraph from the sales report summary>

### Core Pain Points

- <bullet from known_requirements>
- <bullet from known_requirements>
- ...

### Target Outcome

<one paragraph from expected_solution>

## 2. Recommended Solution

### Solution Overview

<one paragraph naming the recommended scenario, confidence level, and current report completeness>

### Product Capability Match

  - Constraint: <constraint>
  - Constraint: <constraint>
- ...

### Technical Flow

- <implementation_note>

## 3. Similar Cases

- **<case_id>** — <industry>: <solution_summary> (Outcome: <outcome>)
- ...

## 4. Assumptions And Open Questions

Open questions before commercial commitment:

- <missing checklist item>
- <missing checklist item>

Subject-consistency reminder: the entity that signs with us must match the WeChat Pay merchant account holder. Do not commit before this is confirmed.
```

## Authoring guidance for the agent

- Keep the `> **Demo data — fictional customer.**` callout for any output generated in offline mode.
- Replace `[Your Company]` with the user's actual company brand only after the user confirms.
- Prefer neutral, non-committal phrasing in section 2 — assumptions live in section 4.
- Cite case IDs by their `case_id` field; never substitute real customer names in the body even if the report mentions one.
- Section 4 must list at least one open question if `completeness_score < 1.0`.
